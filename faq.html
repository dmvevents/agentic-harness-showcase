Flexible Training Plans (FTP)

5 answers

What is the best practice when a Flexible Training Plan (FTP) expires mid-training?

Checkpoint aggressively and plan the run to finish inside the reserved window; an FTP is a fixed-term capacity reservation that does not auto-extend or auto-fall-back to on-demand. When the plan ends, instances are reclaimed, so resume from the last checkpoint on a fresh reservation (a new FTP, a Capacity Block, or on-demand if available).

✓ do: Size the FTP to cover the full training duration plus buffer, and use HyperPod auto-checkpointing so a fresh reservation can resume from the latest checkpoint.

✗ not: Assuming the cluster will silently keep running or auto-switch to on-demand when the plan expires — capacity is reclaimed and the cluster can enter an Updating/RollingBack loop.

⚖ depends: It depends on capacity at expiry: a replacement FTP or Capacity Block in the same AZ may not be available, so on-demand may be the only (and scarcest) fallback for P5/P5en.

src: slack:#sagemaker-hyperpod-interest · conf 0.75

Can a customer extend a Flexible Training Plan once it has started?

No — an FTP is a fixed-term, all-upfront capacity reservation and cannot be extended after it starts. To continue past the end date you purchase a new plan (search-training-plan-offerings → create-training-plan) for the additional period.

✓ do: Plan total duration up front, or buy a second overlapping/back-to-back FTP before the first expires to avoid a capacity gap.

✗ not: Believing you can add weeks to a running plan the way you bump an on-demand instance — extension is not supported.

⚖ depends: The follow-on plan's availability depends on capacity at purchase time and is AZ/region-locked, so a contiguous block may not exist.

src: slack:#hyperpod-gtm · conf 0.8

Can a HyperPod Flexible Training Plan be moved between AWS regions (e.g. us-east-1 to us-east-2)?

No — FTP/HyperPod reserved capacity is region- and AZ-locked and cannot be migrated. To switch regions you terminate the existing reservation and purchase a new plan in the target region, then move data and re-create the cluster setup there.

✓ do: Purchase a fresh plan in the target region and re-run the cluster/lifecycle setup; only the data and configuration need to be copied over.

✗ not: Expecting AWS to transfer an existing committed plan from one region to another — instances cannot be migrated between regions.

⚖ depends: Whether you can re-use prior setup depends on what's region-scoped: lifecycle scripts and IaC are portable, but the capacity commitment itself must be re-purchased and subject to target-region availability.

Why does FTP search return 'Internal account is not allowlisted to search training plan offerings'?

FTP offering search (search-training-plan-offerings) is allowlist-gated on internal/isengard accounts via a feature flag (the LooseLeaf/#ask-looseleaf gate). Customer accounts that already hold an FTP are not gated; the allowlist applies to internal accounts trying to search offerings.

✓ do: For an internal/isengard test account, request the SearchTrainingPlanOfferings allowlist via #ask-looseleaf with account ID and regions.

✗ not: Concluding there is no capacity or the API is broken — the error is an account-allowlist gate, not a capacity or permissions bug.

⚖ depends: Real customer accounts holding an FTP have no gate, so this only blocks internal testing flows; verify which account type you're on before filing tickets.

Can a Flexible Training Plan be shared with or transferred to another AWS account?

Cross-account sharing of FTP is not the same as EC2 Capacity Block sharing and should not be assumed — verify in current docs. The robust path for moving a workload to a new account is typically to re-purchase/re-provision in the target account.

✓ do: Confirm the supported sharing model in current docs; if unsupported, provision the plan in the account that will own the workload.

✗ not: Assuming FTP shares exactly like Capacity Block sharing because they use the same underlying capacity — the sharing/RAM semantics differ.

⚖ depends: Depends on the current feature set; Capacity Block sharing exists, but FTP sharing support has lagged and may differ by region/release.

Capacity & reservations

3 answers

What is the relationship between FTP and EC2 Capacity Blocks, and when use which?

FTP 'uses capacityBlocks under the hood' — same underlying hardware reservation model. EC2 Capacity Blocks are the non-allowlisted, self-service path to the same GPUs (purchase-capacity-block), useful when the internal FTP search is gated or for self-hosted/EC2-native workloads.

✓ do: Use FTP when you want SageMaker/HyperPod-managed capacity with PPA line-item eligibility; use EC2 Capacity Blocks for self-hosted inference/EC2-native workloads or when FTP search is allowlist-gated.

✗ not: Treating FTP and Capacity Blocks as fundamentally different hardware pools — they reserve the same underlying capacity.

⚖ depends: Capacity Blocks have a 24h minimum and are AZ-locked, max ~6 months (no 12-month commit); for 12-month reservations use an ODCR instead, and credits generally can't be applied to all-upfront CB/FTP without a pre-processing step.

src: slack:#ec2-capacity-blocks-interest · conf 0.74

Can a customer mix Capacity Block and Flexible Training Plan capacity in one HyperPod cluster?

This is not a supported, guaranteed mix-and-match in a single cluster's instance group out of the box — capacity sources are reserved and AZ-scoped separately. Treat each reservation as its own capacity pool and verify current support before designing around it.

✓ do: Provision separate instance groups/clusters per reservation source and keep each within its own AZ, rather than assuming one cluster transparently pools CB + FTP.

✗ not: Assuming you can freely blend a Capacity Block and an FTP into one instance group like fungible on-demand capacity.

⚖ depends: Support evolves — confirm against current HyperPod docs/service team, since flexible instance groups and continuous provisioning have been expanding what a single cluster can combine.

src: slack:#sagemaker-hyperpod-interest · conf 0.6

Why does 'insufficient capacity' appear even with an active FTP that has available capacity?

An active FTP guarantees a reservation but instances still launch via EC2 RunInstances into a specific AZ/subnet; a capacity error usually means a subnet/AZ mismatch or transient EC2 capacity at launch. Ensure the cluster subnet is in the same AZ as the FTP reservation.

✓ do: Use a subnet in the exact AZ of the FTP reservation (the purchase warning calls this out) so RunInstances draws from the reserved pool.

✗ not: Assuming an active plan means instances launch from any subnet/AZ in the VPC — the reservation is AZ-pinned.

⚖ depends: If the AZ matches and capacity still fails, it may be a genuine EC2 shortfall needing escalation to the EC2 team rather than a config issue.

Billing & cost

5 answers

Does FTP/HyperPod spend count toward a SageMaker PPA and is it discounted by the PPA/EDP?

FTP spend counts toward PPA attainment and is eligible for cross-service (EDP) discounts only if the private offer has a Flexible Training Plan line item with the specific instance type priced. Without that line item, the consumption is not covered.

✓ do: Add an explicit 'SageMaker HyperPod Flexible Training Plan instances' line item (with instance type) to the private offer so the spend both attains and is discounted by the PPA.

✗ not: Assuming all SageMaker consumption is automatically PPA-eligible — FTP needs the explicit line item, which is why the wiki reads 'yes and no at the same time'.

⚖ depends: Discount stacking order matters: a Savings Plan reduces the on-demand rate first, then the cross-service/EDP discount applies to the already-discounted amount.

src: slack:#ai-ml-sales-specialists-emea · conf 0.72

Are SageMaker HyperPod / FTP / Reserved Instances eligible for MAP 2.0 credits?

MAP 2.0 explicitly excludes 'SageMaker training plans for training jobs or HyperPod clusters', so FTP/training-plan spend does not earn MAP credits. HyperPod Reserved Instance (RI) spend, by contrast, is eligible to earn MAP 2.0 credits.

✓ do: Position HyperPod Reserved Instances (not training plans) when the customer needs the spend to earn MAP 2.0 credits.

✗ not: Treating all SageMaker spend as MAP-eligible because 'SageMaker is included' — the training-plan/HyperPod-cluster carve-out overrides the general inclusion.

⚖ depends: Earned MAP credits (MAP 2.0 or MAP PPA) cannot be spent on future RIs, and Nova/Bedrock model usage has its own eligibility — confirm the specific SKU against current MAP docs.

How does HyperPod usage show up in billing — as EC2 or as SageMaker?

HyperPod launches EC2 (on-demand/reserved, non-spot) instances via EC2 RunInstances and emits SageMaker HyperPod metering records (REMO). The charge surfaces as SageMaker HyperPod usage, not raw EC2, and does not include connected services like EKS, FSx for Lustre, or S3.

✓ do: For EDP/commitment planning, treat HyperPod compute as SageMaker HyperPod line items and budget EKS/FSx/S3 separately as their own charges.

✗ not: Assuming HyperPod compute bills as plain EC2, or that the HyperPod price covers attached EKS/FSx/S3.

⚖ depends: Both EC2 (e.g. spot) and HyperPod metering line items can appear when the service account launches the instance; exact surfacing depends on how the cluster/metering is configured.

What's the best way to get cost savings for HyperPod inference beyond SageMaker Savings Plans?

SageMaker Savings Plans give the headline discount (~64% off on-demand); beyond that, use reserved capacity (FTP-for-inference or Capacity Blocks/RIs) for guaranteed GPUs at committed rates, and right-size with task governance idle-sharing to raise utilization.

✓ do: Combine a Savings Plan with reserved inference capacity and high cluster utilization (idle resource sharing) to lower effective $/GPU-hour.

✗ not: Assuming Savings Plans are the only lever — leaving GPUs underutilized erases the discount's benefit.

⚖ depends: Discount stacking has an order (Savings Plan first, then EDP/cross-service), and not all instance types (e.g. some G5 configs) are supported by every plan — verify per instance type.

Does HyperPod pricing cover the connected services (EKS, FSx for Lustre, S3) a cluster uses?

No — HyperPod pricing covers the cluster compute only and explicitly does not include charges for connected services such as Amazon EKS, FSx for Lustre, or S3. Those are billed separately under their own service line items.

✓ do: Budget EKS control-plane, FSx for Lustre, and S3 as separate line items on top of HyperPod compute when modeling total cost.

✗ not: Assuming the HyperPod per-hour price is all-in and covers storage/orchestration dependencies.

⚖ depends: Which connected services apply depends on the architecture (Slurm vs. EKS, FSx vs. S3-only); a minimal setup may incur few extras while a production one adds AMP, ALBs, etc.

Orchestration & bring-up

4 answers

Should a customer choose EKS or Slurm orchestration for HyperPod custom model training?

Both EKS and Slurm are fully supported and the choice is largely team preference — Slurm for HPC-comfortable teams, EKS for Kubernetes-native MLOps. HyperPod supports both for training and inference.

✓ do: Pick the orchestrator the team already knows: Slurm if they prefer classic HPC scheduling, EKS if they want K8s-native pipelines and inference co-location.

✗ not: Assuming EKS is the only option for fully custom training, or that the two have identical feature sets.

⚖ depends: Feature parity is not guaranteed — historically EKS received new features ahead of Slurm (e.g. continuous provisioning, custom AMI), so check which features you need on each path.

Can HyperPod Slurm clusters specify a minimum capacity requirement?

Yes — HyperPod Slurm clusters with continuous provisioning support specifying minimum capacity requirements, so the cluster won't proceed/scale until at least the minimum healthy nodes are available, then continuously maintains that capacity.

✓ do: Set a minimum capacity requirement with continuous provisioning so the cluster only becomes usable once the minimum node count is healthy.

✗ not: Assuming a cluster always launches with exactly the requested node count regardless of capacity — continuous provisioning fills toward the target over time.

⚖ depends: This is tied to continuous provisioning mode; clusters created without it behave differently, and the minimum-capacity option for Slurm rolled out after EKS.

Why can't a customer provision HyperPod EKS on Kubernetes 1.35 even though the AMI was released?

AMI availability is necessary but not sufficient — the HyperPod API and Terraform/IaC providers must also support the version. If the API and providers don't yet support 1.35, the version is effectively unprovisionable until they're updated, regardless of AMI release.

✓ do: Confirm API + Terraform provider support (not just AMI release) before promising a new K8s version; until then provision the latest fully supported version (e.g. 1.34).

✗ not: Treating an AMI release note as proof a K8s version is usable end-to-end.

⚖ depends: EKS 1.35 also requires Continuous provisioning mode; timeline for API/provider support is WIP and should be verified with the service team.

Among AWS managed Slurm options (ParallelCluster, PCS, HyperPod-Slurm), which fits large GenAI training?

For large-scale distributed GenAI training (1000+ GPU clusters) with resilience needs, HyperPod-Slurm is the purpose-built choice — its core value is automatic fault tolerance (auto node replacement, auto-resume, deep health checks, checkpointless training). ParallelCluster/PCS are classic HPC orchestration without HyperPod's managed resiliency.

✓ do: Position HyperPod-Slurm for resilient large-scale model training; reserve ParallelCluster/PCS for traditional HPC where managed GPU fault tolerance isn't the differentiator.

✗ not: Treating the three as interchangeable — the lines blur on scheduling, but only HyperPod brings managed GPU resiliency for long training runs.

⚖ depends: PCS is adding GenAI-oriented features (e.g. deep health checks), so the boundary is narrowing; pick based on the customer's resiliency and managed-service requirements.

Resilience & recovery

4 answers

How does HyperPod handle a failed GPU/node during training?

HyperPod runs automatic instance replacement: the health-monitoring agent continuously checks node/GPU health (labeling fault-types) and, combined with auto-resume and auto-checkpoint, replaces faulty nodes and resumes training without manual intervention.

✓ do: Rely on the health-monitoring agent + auto-resume/auto-checkpoint so a detected faulty GPU triggers automatic node replacement and training resumes from checkpoint.

✗ not: Using trouble-ticket counts as the source of truth for node failures — HyperPod replaces instances automatically, so tickets undercount real fault events.

⚖ depends: If replacement stalls it's often an EC2 capacity shortage (no healthy node to swap in), which needs EC2 escalation rather than a resilience-agent fix.

src: slack:#hyperpod-ops · conf 0.8

Do EKS/HyperPod node health checks catch GPU-level faults like hung ranks or NCCL deadlocks?

No — EKS and HyperPod node/health checks operate at the node level and cannot see in-process GPU faults such as hung ranks, straggler GPUs, or NCCL deadlocks. Sub-process fault detection needs a tool like NVIDIA Resiliency Extension (NVRx) with fast heartbeat/straggler detection.

✓ do: Layer NVRx (or equivalent) on top of HyperPod auto-resume to get sub-5-second heartbeat/straggler detection for in-process GPU faults.

✗ not: Assuming node-level health checks and kubelet probes are sufficient to detect a hung rank or NCCL deadlock inside the training process.

⚖ depends: HyperPod auto-resume handles node/hardware faults well; the gap is specifically intra-process GPU faults, so the extra tooling only matters for long multi-node runs sensitive to stragglers.

src: slack:#hyperpod-ops · conf 0.75

What is HyperPod's core value proposition for distributed training?

HyperPod is a managed resilient cluster: it auto-checkpoints, auto-resumes, and automatically replaces faulty nodes on instability or crashes, so long multi-node training survives hardware faults with minimal lost work. Resiliency — not raw provisioning — is the differentiator.

✓ do: Lead with resiliency (auto node replacement + auto-resume + checkpointing, plus checkpointless/elastic training) when justifying HyperPod over self-managed clusters.

✗ not: Pitching HyperPod purely as a way to get GPUs — that undersells the managed fault tolerance that's the actual value.

⚖ depends: Resilience covers node/hardware faults automatically; in-process GPU faults (hung ranks, stragglers) still need add-on tooling like NVRx, and replacement can stall on EC2 capacity shortages.

src: slack:#hyperpod-ops · conf 0.78

What does the HyperPod 'health monitoring agent' do and which orchestrators support it?

The HyperPod health monitoring agent runs continuous health checks on cluster instances, automatically detecting faulty hardware (e.g. bad GPUs) and triggering node replacement. It's GA for both EKS and Slurm-orchestrated clusters, and is complemented by on-demand deep health checks.

✓ do: Enable the health monitoring agent on Slurm or EKS clusters and use on-demand deep health checks to proactively verify GPU health on running instances before a big run.

✗ not: Assuming automated health monitoring is EKS-only — it reached GA for Slurm clusters as well.

⚖ depends: The agent acts at the node/hardware level; pair it with deep health checks for proactive GPU verification and with in-process tooling for rank-level faults.

Networking & EFA

1 answer

What does HyperPod support for EFA-only network interfaces mean, and how many can an instance have?

HyperPod now supports EFA-only network interfaces (no traditional ENI/ElasticNetworkInterface) per instance group, dedicating network cards to RDMA. The per-instance EFA count is bounded by the instance's network cards — you can assign only one EFA or EFA-only interface per network card.

✓ do: Use EFA-only interfaces on instances with many network cards (e.g. P6-B200 has 8) to dedicate each card to an EFA device for max RDMA bandwidth.

✗ not: Assuming you can attach unlimited EFA devices — the limit is one EFA/EFA-only interface per network card.

⚖ depends: Limits differ by instance: on P5/P5e each ENI supports MaximumNetworkInterfaces of 2 (device indices 0 and 1 only), and the primary interface already uses DI=0; for EFA-only the per-card rule dominates.

Does FTP capacity support inference endpoints, not just training?

Yes — since the Nov 2025 launch, Flexible Training Plans support inference endpoints, giving guaranteed GPU capacity for planned evaluations and production peaks. You reference the FTP reservation when creating the endpoint so it draws from reserved capacity.

✓ do: Reserve FTP capacity sized for the inference peak and point the endpoint at the plan to get guaranteed GPUs instead of competing for on-demand.

✗ not: Assuming FTP is training-only; older docs/mental models predate the inference support and lead people to buy separate on-demand capacity.

⚖ depends: Detailed configuration docs are thin — verify the current API/flow in docs, since the exact way to attach the FTP ARN to an endpoint has changed across releases.

What's the difference between InferenceEndpointConfig and JumpStartModel in the HyperPod managed inference operator?

InferenceEndpointConfig deploys cluster-native models and draws from the cluster-usage quota, while JumpStartModel deploys JumpStart models and draws from an endpoint-usage quota that defaults to 0. A new JumpStart deployment commonly fails until that endpoint-usage quota is raised.

✓ do: For JumpStart-based endpoints, request an endpoint-usage quota increase first (default 0); use InferenceEndpointConfig for cluster-native models on cluster-usage quota.

✗ not: Deploying a JumpStartModel and assuming it will schedule on the default quota — endpoint-usage quota of 0 blocks it.

⚖ depends: Accounts with a low C-score (<300) need a quota-increase ticket regardless of path; the right CRD depends on whether the model is cluster-native or from JumpStart.

Task governance

2 answers

What does HyperPod task governance use to enforce compute quotas and scheduling?

HyperPod task governance is built on Kueue for fine-grained compute/memory quota allocation, gang scheduling, cluster policies, and idle resource sharing (borrowing unused quota). Pods awaiting admission/topology gates show as SchedulingGated until the gates lift.

✓ do: Use task governance fine-grained quotas plus idle resource sharing so teams can borrow unallocated capacity beyond their guaranteed quota when the cluster is idle.

✗ not: Reading a SchedulingGated pod as a failure — it's Kueue holding the pod until admission/topology gates clear, which is expected behavior.

⚖ depends: Gang scheduling and borrowing depend on cluster policy configuration; if no quotas/policy are set, behavior differs from a fully governed cluster.

src: slack:#ask-hyperpod · conf 0.75

Why can't I delete a HyperPod cluster — it's blocked by task governance policies?

Task-governance admission/quota objects can block cluster deletion. The fix is to remove the governing resources (compute quotas, cluster policy, Kueue objects) first, or address the admission webhook, then delete. A cluster reporting governance blocks may still need the governance add-on/CRDs cleaned up even if no quotas appear set.

✓ do: Drain/clear task-governance quotas and policies (and any lingering webhook/CRD) before issuing the cluster delete.

✗ not: Assuming a cluster 'without task governance installed' can't be governance-blocked — residual CRDs/webhooks can still hold the delete.

⚖ depends: The exact blocking object varies (compute quota vs. cluster policy vs. admission webhook); inspect each before forcing deletion.

Is custom AMI supported for HyperPod, and on both Slurm and EKS?

Custom AMI is supported for HyperPod EKS but only with Continuous Scaling/provisioning mode enabled. Custom AMI support for HyperPod Slurm has lagged and may not be available — verify in current docs.

✓ do: For a custom AMI on HyperPod EKS, create the cluster with continuous provisioning mode enabled (now the recommended default for EKS).

✗ not: Assuming the EKS custom-AMI blog applies equally to Slurm, or that custom AMI works without continuous provisioning.

⚖ depends: The HyperPod AMI is built on DLAMI (which itself bases on public EKS AMIs); 'Public AMI releases' vs. Slurm/EKS AMI releases differ, and CMK/encryption scope-down to a specific instance group has known limitations.

Does a HyperPod cluster support multiple instance groups / instance types?

Yes — HyperPod supports multiple instance groups, and flexible instance groups (launched Apr 2026) expand mixing of instance types within a cluster. Different groups can serve training vs. inference or different GPU generations.

✓ do: Use separate (or flexible) instance groups to run mixed instance types or to dedicate groups to training vs. inference within one cluster.

✗ not: Assuming a cluster is locked to a single instance type, or that a RIG (restricted instance group) supports arbitrary multiple types out of the gate.

⚖ depends: Support for multiple/flexible instance types is a relatively recent capability; older clusters or modes may not have it, so verify the cluster's provisioning mode and release.

What is the smallest viable HyperPod cluster, and what blocks add-ons like observability from installing?

You need at least one healthy node before installing add-ons such as the HyperPod observability add-on, and the node must be large enough to fit the add-on's pods. Add-on install failures on small/empty clusters are usually insufficient node capacity to schedule the collector pods.

✓ do: Provision at least one sufficiently large node first, then install the observability/OTel add-on so its collector pods can schedule.

✗ not: Installing cluster add-ons against an empty or too-small cluster and reading the scheduling failure as a broken add-on.

⚖ depends: On very large clusters (800+ nodes) the opposite problem appears — add-on deploys can time out from high metric volume to AMP, needing tuning rather than more capacity.

Ways of procurement

FTP · ODCR · Capacity Blocks

The three instruments for reserving accelerated compute, mined from real procurement discussions. Each fits a different term and billing shape — picking the wrong one is a common, expensive mistake.

InstrumentTerm it servesBillingKey constraint
EC2 Capacity Block (CB)short — up to 182 days (26 weeks), startable up to 8 weeks outall-upfront at purchase; ~40–50% under on-demand for the windownot 12 months; no team cost-tagging in CUR 2.0; no refund
ODCR (on-demand capacity reservation)open-ended hold in a specific AZ, cancel anytimefull on-demand rate whether running or notopportunity cost if idle; some C-ODCRs can't be modified
SageMaker FTP (Flexible Training Plan)multi-month, SageMaker-managed (the CB equivalent for HyperPod/training)all-upfrontallowlist-gated; can't be extended once started (as of 2026-06)
The 12-month trap. Capacity Blocks max out at 182 days — a customer asking for a 1-year commit needs an ODCR + Savings Plan (AWS-assisted) or a 6-month CB/FTP with intent to extend. Promising a 1-year self-serve CB is the most common procurement mistake.

Procurement gotchas

Bugs and traps worth knowing

CB price locks at purchase

A Capacity Block's price is set at time of purchase, not start date. Later price changes apply to new offerings only.

✗ don't assume the price floats to the cheaper start-date rate

CB can't be tagged in CUR 2.0

Capacity Blocks can't be tagged to teams/cost centers in CUR 2.0 — team-level cost attribution is blocked.

✗ don't promise per-team CB chargeback via tags

No CB ↔ FTP portability

An EC2 Capacity Block can't be converted to a SageMaker FTP or vice versa — they're separate reservations.

✗ don't plan to migrate a CB into HyperPod FTP later

HyperPod update loop on plan expiry

A HyperPod cluster can get stuck in an Updating/RollingBack loop if a training plan expires while an instance group still references it.

✗ don't let a plan lapse with live instance groups attached

CB launch can fail despite an active reservation

A paid, active Capacity Block can still fail to launch instances — verify the launch, don't assume upfront payment guarantees capacity.

✗ don't treat "reservation active" as "instances running"

All-upfront breaks credit application

FTP/CB are all-upfront; the zero-amount invoice won't apply credits without special pre-processing.

✗ don't assume MAP/promo credits auto-apply to an upfront CB
