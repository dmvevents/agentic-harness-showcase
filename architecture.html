Architecture

Tool, Agent, Repo, Portal

Four layers with one trust boundary. The deterministic tool and the version-controlled repo are the source of truth; the agent orchestrates; the portal presents. The public tier never touches sensitive data.

PORTAL — presentation
Public tier (this static site): landing, directional calculator, this page.  |  Restricted tier planned v2: auth + RBAC + audit, catalog browse, report download.
↓
AGENT — conversational orchestrator
Slot-fills inputs, asks follow-ups, routes to a tool function, formats the structured result. Never invents a formula or a benchmark number. Cannot mutate.
↓   structured call
TOOL — deterministic functions
discover_capacity · classify_failure · plan_cluster · ask_hyperpod · taxonomy · size_cluster.  Same input → same output. The failure classifier is a compiled regex table, not a prompt.
↓   reads from
REPO — single source of truth (git, PR-gated)
316 instincts · 52 detect→fix patterns · 30 Q&A entries · v2 benchmark entries (config + results + provenance + validation status).
The boundary that matters. The agent surface advises: plan, classify, discover, answer. Spending money (capacity purchase) and mutating a cluster (fix execution) stay in-process, behind a human gate — never exposed on the public MCP/agent surface. A build-failing test enforces that no mutating verb is ever added to the MCP tool catalog.

The Repo planned v2

Every benchmark carries its provenance

A benchmark number with no commit, no image digest, and no validation status is a rumor. Each entry is one PR-contributed JSON document, gated by a CI validation check before merge.

// benchmarks/<id>.json  — one real run, PR-gated
{
  "config": { "model": "llama-3-8b", "workload": "inference",
              "gpu_type": "H100", "instance_type": "p5.48xlarge",
              "gpus_total": 16, "parallelism": {"tp":8,"pp":1,"dp":2},
              "precision": "bf16", "framework": "vllm" },
  "results": { "metric": "tokens_per_sec", "value": 0.0, "unit": "tok/s" },
  "provenance": { "commit": "<sha>", "image_digest": "sha256:...",
                  "command": "redacted-safe", "audit_ref": ".claude-sessions/<run>/" },
  "validation": { "status": "validated", "validator": "<reviewer>",
                  "gates_passed": ["fabric-active","no-tcp-fallback","reproduced-2x"] }
}
Separation of authoring and approval. A contributor cannot self-validate their own benchmark entry — the same two-pass discipline the harness already uses for instincts. The CI gate also runs a sensitive-data linter (no customer names, no account ids, no internal hostnames) before merge.

Reporting

Two artifacts from one structured result

customer-safe

Executive summary

Prose, no internal pricing or account ids. Every quantitative claim is labeled either "validated benchmark" (backed by a repo entry) or "directional estimate" (from the public-spec calculator).

reproducibility backing

Technical artifact

The exact config, commit, image digest, and command, so a third party can re-run it and get the same number. The proof behind the summary.

MVP renders both inline (Markdown/HTML). v2 persists them to S3 with a signed download link and writes an audit record.
