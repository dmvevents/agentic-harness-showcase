a source of truth, not a slide deck

Benchmark intelligence,
with provenance.

Every number a customer sees is one of two things, and it is always labeled: a measured result that came off real hardware and through a review gate, or an estimated figure the deterministic calculator projected from public specs. The portal is the machinery that keeps those two honest — a PR-gated repo of benchmark artifacts, a publication state machine, and an agent that retrieves numbers by calling tools rather than guessing them.

The source of truth

A version-controlled benchmark repo, PR-gated

Benchmarks live as immutable, reviewed artifacts in the repository — not in a spreadsheet, a deck, or a model's memory. A number becomes citable only after it passes schema validation, a review gate, and a second-person sign-off.

one run = one artifact

Three files per benchmark

Each result is a directory of three JSON files validated against a locked schema: config.json (what was run), results.json (the metrics), and metadata.json (who, what hardware, what provenance, what state).

immutable once published

The id is a contract

A benchmark_id like <model>-<instance>-<workload>-<yyyymmdd>-<hash> is frozen at publish. A re-run is a new artifact that supersedes the old one — history is never overwritten.

review-gated entry

The PR is the gate

Nothing self-publishes. A result enters through a pull request that the schema check, the consensus gates, and a human reviewer must clear. The author and the validator must be different people.

Why a repo and not a dashboard. A benchmark you can't reproduce is an anecdote. Pinning every number to a commit, an image digest, and a run command means a customer — or a skeptical engineer — can re-run it and get the same answer. The repo is the audit trail.

The artifact schema

The shape of a benchmark

The schema is written once and locked. Below is the metadata.json shape, shown with the kind: "example" template — a schema illustration whose every metric value is 0.0 and is never a real measurement.

metadata.json — the example template 
// kind:"example" — schema illustration, NOT a measurement
{
  "benchmark_id": "<model>-<instance>-training-<date>-example",
  "workload": "training",
  "model": { "name": "example-model", "params_b": 0 },
  "hardware": { "gpu": "H100", "instance_type": "<instance>",
                "node_count": 8, "gpus_total": 64 },
  "technique": { "framework": "<fw>", "precision": "bf16" },
  "kind": "example",        // measured | estimated | example
  "provenance": { "commit": "0000000",
                  "measured_at": "<date-time>" },
  "validation": { "status": "example" },
  "messaging": { "customer_safe": false }
}
what each block guarantees

kind — the single field that says whether this is a real run (measured), a projection (estimated), or a schema template (example, never customer-facing).

provenance — the commit, optional image digest, a redaction-safe command, and the timestamp. This is what makes a result reproducible.

validation.status — where the artifact sits in the state machine below.

messaging.customer_safe — defaults to false. A number is not shareable until a human explicitly flips it, after the result is published.

The example is a guardrail, not data. The locked schema ships one worked artifact with kind:"example" and every metric at 0.0. It exists so tools and tests have a fixture to validate against — and so no one can mistake a placeholder for a benchmark.

The distinction that matters

Measured vs. estimated — never blurred

Both kinds are legitimate. What is forbidden is letting one masquerade as the other. The badge color tells you which you're looking at, every time.

kind: "measured"
measured  came off real hardware

A number observed on an actual run, pinned to a commit and image digest, that cleared the review gate.

  • per-metric kind: "measured" in results.json
  • carries a snapshot_id — a deterministic hash of the inputs
  • only trustworthy if the fabric check passed (no silent TCP fallback)
  • citable as a validated benchmark
kind: "estimated"
estimated  projected from public specs

A directional figure the deterministic calculator derived from published hardware specs and the request — no GPU was touched.

  • per-metric kind: "estimated" — same schema, different label
  • deterministic: same input always yields the same output
  • good for sizing and ballpark cost; not a performance claim
  • presented as a directional estimate, always
One field, two colors. A green measured badge and an amber estimated badge are driven by the artifact's own kind — the portal can't render a measured-looking number from estimated data. The DEGRADE classifier guards the other failure mode: a measured number captured while NCCL silently fell back to TCP is flagged before it can be published as "real."

The lifecycle

The publication state machine

An artifact's validation.status advances along a fixed path. A number is only customer-citable once it reaches published — and it can later age out.

draft
An artifact exists on a branch. Not yet schema-checked, not citable.
↓
schema-ok
Validates clean against the locked metadata / results / config schema.
↓
in-review
Open in a PR. A reviewer who is not the run operator inspects it — author ≠ validator is enforced.
↓
validated
Consensus gates and the human reviewer signed off; gates_passed records which.
↓
published
Merged, immutable, citable. Only now can customer_safe be flipped to true by a human.
↓   later
stale  |  superseded
stale: the world moved (new driver, new image) and the number needs a refresh. superseded: a newer artifact supersedes this one. Neither deletes history.

The number-provenance guard

The agent retrieves numbers — it never invents them

A report reads like prose, but no quantitative claim in it originated in the language model. The agent fills slots and formats; deterministic tools supply every number, and each number arrives already labeled with its kind and provenance.

1 · the ask
"How does a 70B model train on a p5 cluster?" → the agent parses the request into a benchmark selector (model, workload, hardware).
↓
2 · CALL query_benchmark(selector)
A deterministic tool searches the repo for published artifacts matching the selector and returns the artifacts verbatim — values, units, and kind intact. If nothing matches, it returns nothing.
↓   matched artifacts (or none)
3 · fall back to estimate, labeled
No measured match? The deterministic calculator returns an estimated figure from public specs — same input, same output — tagged so it can never be mistaken for a measurement.
↓   tool outputs only
4 · CALL generate_report(artifacts)
The report builder renders prose around the tool outputs. The model never types a benchmark digit — it arranges numbers the tools already produced, each with its measured / estimated tag.
↓
5 · the report
Every figure links back to its artifact id, commit, and kind. A reader can trace any claim to a repo entry — or see plainly that it was a directional estimate.
The guard, concretely. The report builder accepts numbers only from query_benchmark and the deterministic estimator; a free-form number in model output is not a permitted source. If a question has no published artifact and no estimable answer, the honest output is "no benchmark on file for that configuration" — the same refusal discipline the conversational layer uses for off-corpus questions. A made-up number is a bug, not a fallback.
