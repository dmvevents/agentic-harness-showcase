How it actually works,
layer by layer.

Beneath the three lanes is a layered optimization framework, a self-curating knowledge store, and an autonomous control loop. This page is the deeper story: what each layer covers, how the meta-learner decides what's relevant, and how the agent stands up infrastructure, deploys a POC, and answers a customer from a version-controlled corpus — not from a model guessing.

The optimization stack

Twelve layers, L0 → L9

A GPU cluster fails — and improves — at every layer from the CUDA kernel to the operational knowledge that codifies what went wrong. The agent maps each layer to a specific optimizer, with an honest status: not everything is built, and the framework says so.

Why a layer model matters to a customer. When a benchmark is slow or a bring-up hangs, the question is always which layer. A TCP fallback is L5 (fabric). A bad NCCL algorithm is L2 (framework). A missing kernel module is L6 (OS). The agent classifies the failure to a layer first, then reaches for that layer's optimizer — instead of guessing.

The meta-learner

A self-curating knowledge store, not a static FAQ

The corpus is 316 atomic instincts. Each is a structured fact: a rule, its evidence, a confidence, a positive and a negative example, and an applies_to qualifier that bounds where it's true. The meta-learner is how the agent retrieves the right few for the task in front of it.

the atomic instinct 
// one instinct — structured, bounded, attributed
{
  "id": "K0014",
  "kind": "gotcha",
  "rule": "NCCL silently uses TCP if EFA
           plugin isn't advertised",
  "applies_to": { "substrate": ["efa_srd"],
                  "layer": ["L5_efa_fabric"] },
  "confidence": 0.92,
  "positive_example": "...",
  "negative_example": "...",
  "source": "verified on p5.48xlarge"
}
retrieval — two-stage filter

1. applies_to gate. Given the project profile (substrate, layer, workload, instance type), only instincts whose qualifier matches that profile survive. EFA-only rules never leak into a TCP or single-host project. Universal rules (empty qualifier) always pass.

2. Keyword scoring. Score the survivors by overlap between the task description and each instinct's rule / evidence / source. Ties break on confidence.

Return the top-n (default 3), each carrying a ≤40-char negative_rule_short to prime the prompt: "don't do X."

The coverage ratchet. Every instinct must stay retrievable (reachable by phase or layer), actionable (has a positive and a negative example), and attributed (has a source). The floor is 100% / 100% / 100%, enforced by a test and a pre-commit hook — it never lowers. Raw instinct count is treated as a vanity metric; coverage is the real number.

How the store grows itself

New knowledge enters through a disciplined, review-gated pipeline — never by overwriting the canonical corpus directly:

Authoring and approval are separate passes — a miner proposes, a reviewer validates. A candidate cannot self-promote, and the ratchet refuses any instinct that isn't retrievable, actionable, and attributed.

Autonomous orchestration

Goal sentence → running infrastructure

The orchestrate lane turns one sentence into an ordered plan, then drives it through an empirical ladder that observes, diagnoses, fixes, and retries — the same loop a human SRE runs, made deterministic.

1 · PARSE the goal
"16-node HyperPod EKS p5en, NCCL test" → {instance_type, node_count, workload, gates}.
↓
2 · SYNTHESIZE the plan
Resolve catalog modules, order by dependency (vpc → eks → cluster → efa-validate → workload), and validate every emitted manifest against the instance-type DB — the B200 8-vs-16 EFA-NIC trap is caught here, before any apply.
↓   ordered modules
3 · EMPIRICAL LADDER (per module)
apply → observe output → classify the failure (52-pattern regex table) → rank by severity (BLOCK > DROP > DEGRADE > WARN, deterministic id tie-break) → dispatch the codified fix → retry ≤ N. Every attempt's log signature + matched instinct is recorded.
↓   on success
4 · AUDIT + LEARN
A redacted audit trail per run; a new failure→fix that wasn't in the corpus becomes a mined candidate for the meta-learner. The loop feeds the store that feeds the loop.
Determinism is the safety property. The same log signature always ranks to the same fix — no coin-flip between equal-severity matches, so no wasted retry. A BLOCK severity (e.g. EFA device absent) stops the run rather than reporting a meaningless number. Hard guardrails (anything that could be unsafe) get exactly one attempt, never a retry loop.

From ask to POC

Standing up a proof-of-concept, autonomously

The end-to-end story a solution architect cares about: a customer asks for a capability; the agent scopes it, finds capacity, stands up the infrastructure, validates the fabric, runs the workload, and hands back a report — with a human gate on anything that spends money or mutates a cluster.

1 · scope & size

Understand the ask

The conversational layer slot-fills the requirement (model, workload, scale, region, deadline) and calls the deterministic sizing tool for a directional footprint, parallelism, and cost.

2 · capacity

Find the hardware

Read-only discovery of EC2 Capacity Block / FTP offerings — the right instance type, region, and duration. Purchase stays behind the human-gated spend firewall.

3 · provision

Stand up the cluster gated

Synthesize the bring-up plan (VPC → EKS → HyperPod → nodegroup), apply through the empirical ladder, self-healing known failures as it goes.

4 · validate fabric

Prove the network is real

Run an unprivileged NCCL test; the classifier confirms RDMA is active and NCCL didn't silently fall back to TCP — so the number means something.

5 · run the workload

Deploy the POC

Deploy the training / inference / disagg workload onto the validated cluster — the same manifests the catalog modules emit.

6 · report & tear down

Hand back proof

An executive summary backed by a reproducible technical artifact (config + commit + image digest). Scale to zero and release capacity when done.

Where the gate sits. Steps 1, 2, 4, 6 are read-only or reversible and run autonomously. Steps 3 and 5 mutate real infrastructure and cost money — they execute only under the live-cluster gate, and a capacity purchase additionally requires the 3-factor firewall the agent cannot satisfy on its own. The agent talks back

Answers from a corpus, not a hunch

The customer-facing surface is conversational, but it is structurally constrained: it can only return what a deterministic tool or the corpus gave it. It is incapable of inventing a benchmark number or a formula.

customer asks 
"Will a 70B model fit on two
 p5 nodes for fine-tuning, and
 how is HyperPod billed?"
agent answers — grounded

Sizing → calls size_cluster: directional footprint exceeds two nodes' usable VRAM → over, suggests more nodes or quantization. (labeled directional, not a benchmark.)

Billing → calls ask_hyperpod: returns the curated KB answer with a do / do-not / depends tier, a source, and a confidence — or says "not in corpus" rather than guessing.

The contract. Every quantitative claim the agent makes is tagged at the source: "validated benchmark" (backed by a PR-gated repo entry with provenance) or "directional estimate" (from the public-spec calculator). If the corpus doesn't cover the question, the honest answer is "I don't have that" — which is itself a feature.
