agentic-harness is an autonomous agent for AWS GPU cluster operations. The hard part isn't the chat — it's that every number and every fix comes from a deterministic tool and a version-controlled corpus, not from a language model guessing. Same input, same output. The pattern

Tool · Agent · Repo

Three layers, one boundary: the language model never invents a formula or a benchmark number. It collects inputs, routes to a deterministic function, and formats the structured result.

deterministic

The Tool

Pure, structured functions: capacity discovery, failure classification, plan synthesis, sizing math. A compiled 52-pattern regex table — not a prompt — maps a log signature to a codified fix. Same input → byte-identical output.

conversational

The Agent

A slot-filling orchestrator. It asks follow-up questions, picks the right tool function, passes structured arguments, and renders the result into prose. It cannot emit a number a tool didn't return, and it cannot mutate a cluster or spend money.

source of truth

The Repo

One git repository: 316 atomic instincts, 52 detect→fix patterns, a 30-entry HyperPod Q&A KB — and (v2) real benchmark runs, each carrying its commit, image digest, and validation status. PR-gated.

Coverage, not count. Raw instinct count is a vanity metric. Every instinct is retrievable (by phase or layer), actionable (has a positive and negative example), and attributed (has a source). That floor never lowers — it's enforced by a test and a pre-commit hook.

The interface

Three lanes: ask · capacity · orchestrate

ask

Answer

Route a HyperPod question to the curated KB. Returns the answer with do / do-not / depends tiers, a source, and a confidence — never an unsourced guess.

cluster-ops ask "how is hyperpod billed"
capacity

Provision

Read-only discovery of EC2 Capacity Block / Flexible Training Plan offerings. Purchase is non-refundable and sits behind a 3-factor spend firewall the agent cannot satisfy alone.

cluster-ops capacity discover \
  --instance-type p5.48xlarge --count 2
orchestrate

Fix

Goal sentence → ordered module plan → the empirical ladder: apply, observe, classify the failure, match an instinct, dispatch the fix, retry. Catches hardware traps at plan time.

cluster-ops orchestrate \
  "2-node HyperPod EKS p5en, NCCL test" \
  --mode stage

The differentiator

Platforms advise. This agent executes — safely.

Diagnostic advice is becoming a commodity. The value is provable, account-specific remediation reliability: detect a known failure signature, match a codified fix, run it with an audit trail and a spend firewall.

Deterministic, by example

Paste a log line, get a codified fix

The failure classifier is a compiled regex table. The same log always maps to the same instinct, the same severity, and the same fix. No model in the loop.

input — a log signature 
NCCL INFO NET/Socket : Using
  [0]eth0 ; busbw 11 GB/s
output — structured, deterministic

DEGRADE   instinct K0014

NCCL fell back to TCP/sockets — the fabric path is inactive, so the bandwidth number is meaningless.

fix: verify the EFA device plugin is advertising and the pod requests vpc.amazonaws.com/efa; without these NCCL silently uses TCP.

Ecosystem

It plugs into the AWS agent ecosystem — both directions

inbound — others call us

MCP server

Ships an MCP server (cluster-ops-mcp) so Amazon Q Developer, Bedrock AgentCore, and Quick Suite can call our tools. The surface is structurally non-mutating — a test fails the build if a purchase or execute verb is ever added.

outbound — we call AWS

MCP client

A full MCP client into the awslabs servers: aws-api, eks, cloudwatch, pricing, and aws-knowledge — each restrictable to read-only verbs.

Safety boundary. The conversational/agent surface advises: plan, classify, discover, answer. Spending money and mutating a cluster stay behind a 3-factor firewall and a human gate, in-process — never on the public MCP surface. See the architecture →

Honesty guardrails

What this site does not claim

  • The public calculator gives directional, public-spec estimates only — clearly labeled. It is not a benchmark.
  • Real benchmark numbers live in a PR-gated repo behind a validation gate. They carry a commit, an image digest, and a validation status. None appear on this public site.
  • Live multi-node GPU bring-up is wired and reachable, gated on booked capacity — not claimed as run-end-to-end here.
  • The benchmark catalog, restricted portal (auth + RBAC + audit), and S3 report storage are planned v2, described in the design doc.
